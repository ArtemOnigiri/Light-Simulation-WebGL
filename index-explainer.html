How the Light Simulation Works

A deep dive into the wave-equation cellular automata, chromatic dispersion, and GPU ping-pong rendering that produces a rainbow from a prism.

↓ Scroll to explore
01 — Overview

The Big Picture

This simulation doesn't ray-trace light — it actually solves the 2D wave equation on a grid, treating each pixel as a tiny oscillating cell. White light enters as a wave, hits a triangular prism that slows it down, and different color channels (R, G, B) travel at slightly different speeds inside the prism, causing them to separate into a rainbow.

Every animation frame, the wave shader runs 8 simulation steps (4 × ping-pong = 8 updates), then the result is rendered to the screen. All computation happens on the GPU via WebGL2 fragment shaders.

02 — The Grid

Cellular Automata Grid

The simulation domain is a 900 × 600 pixel grid. Each pixel is a "cell" in the automaton. Every cell stores three values, packed into GPU textures:

〰️

Amplitude amp

The current displacement of the wave at this cell. Stored as an RGB vec3 — one independent wave per color channel (Red, Green, Blue).

⚡

Velocity vel

How fast the amplitude is changing. This is the time-derivative of amplitude. Also an RGB vec3.

💡

Accumulated Exposure acc

A running sum of |amplitude| over time. This is what you actually see — it acts like a long-exposure photograph of the wave energy.

🔷

Wall / Speed Map wall

The speed of propagation at this cell. 1.0 = vacuum (air), 0.6 = inside the prism (glass). Set once at initialization.

💡 Key Insight Each cell doesn't just store a single number — it stores three independent waves (R, G, B) simultaneously. This is what makes chromatic dispersion possible.

How Textures Map to the Grid

Each of these values is stored in a 32-bit floating-point RGBA texture (RGBA32F). The grid looks like this:

Texture Channel R Channel G Channel B Size
textureAmp Red wave amplitude Green wave amplitude Blue wave amplitude 900×600
textureVel Red wave velocity Green wave velocity Blue wave velocity 900×600
textureAcc Red accumulated Green accumulated Blue accumulated 900×600
textureWalls Speed (0.6 or 1.0) unused unused 900×600
03 — The Prism

Drawing the Wall Map

The original-shader runs once at startup. Its only job: compute a signed-distance-field (SDF) triangle and write the speed map.

// SDF of an equilateral triangle
float triangle(vec2 p) {
  const float k = sqrt(3.0);
  p.x = abs(p.x) - 1.0;
  p.y = p.y + 1.0 / k;
  if(p.x + k * p.y > 0.0) ...
  return -length(p) * sign(p.y);
}

The SDF returns a negative value inside the triangle and positive outside. This is then converted to a speed value:

Speed Map Rule speed = (d < 0) ? 0.6 : 1.0

The constant INDEX_OF_REFRACTION = 3/5 = 0.6 means waves travel at 60% speed inside the prism. This is the base speed — each color channel gets a slightly different modification on top of that.

04 — The Wave Equation

How Each Cell Updates

This is the heart of the simulation. Every cell updates using a discrete version of the 2D wave equation, which is essentially the physics of ripples:

Discrete 2D Wave Equation (per cell, per frame) force = ampUp + ampRight + ampDown + ampLeft

vel += (force × 0.25 − amp) × speed

amp += vel

Step-by-Step Breakdown

1

Sample the 4 Neighbors

For each cell, we read the amplitude of the 4 directly adjacent cells (Von Neumann neighborhood). The offset between cells is 1.0 / resolution in texture coordinates.

vec3 offset = vec3(vec2(1.0) / u_resolution, 0.0);
vec3 ampUp = texture(u_textureAmp, v_texcoord + offset.zy).rgb;
vec3 ampRight = texture(u_textureAmp, v_texcoord + offset.xz).rgb;
vec3 ampDown = texture(u_textureAmp, v_texcoord - offset.zy).rgb;
vec3 ampLeft = texture(u_textureAmp, v_texcoord - offset.xz).rgb;
2

Compute Force (Discrete Laplacian)

The sum of the 4 neighbors minus 4× the center is the discrete Laplacian — it measures how different this cell is from its surroundings. Here it's written as force × 0.25 − amp, which is equivalent to (neighbors_sum − 4 × amp) / 4.

The Laplacian ∇²amp ≈ (ampUp + ampDown + ampLeft + ampRight − 4 × amp) / 4
3

Scale by Speed (Refraction!)

The Laplacian is multiplied by speed — the wall map value. This is how the prism slows down waves. Inside the prism, speed = 0.6, so the wave propagation is slower, bending the wavefronts.

vel += (force * 0.25 - amp) * speed;
4

Update Amplitude

Velocity is added to amplitude — the cell oscillates. This is the classic Verlet-style integration.

amp += vel;
5

Accumulate Exposure

The absolute value of amplitude is added to the exposure buffer each frame, creating a long-exposure photograph effect:

acc += abs(amp) * 0.002; // ACCUMULATED_EXPOSURE
05 — Chromatic Dispersion

How Colors Separate

This is where the magic happens. Each R, G, B channel represents a different "frequency" of light. The trick: inside the prism, each channel propagates at a slightly different speed.

const vec3 COLOR_SHIFT = vec3(0.002, 0.0, -0.004) * 10.0;
// = vec3(0.02, 0.0, -0.04)

speed += COLOR_SHIFT * smoothstep(0.91, 0.9, speed);

The smoothstep(0.91, 0.9, speed) returns 1.0 when speed < 0.9 (inside the prism where speed = 0.6) and 0.0 when speed = 1.0 (outside). So COLOR_SHIFT is only applied inside the prism.

Red Channel
speed = 0.6 + 0.02 = 0.62

Fastest in glass → least refraction → bends least

Green Channel
speed = 0.6 + 0.00 = 0.60

Baseline speed → medium refraction

Blue Channel
speed = 0.6 − 0.04 = 0.56

Slowest in glass → most refraction → bends most

💡 This matches real physics! In real glass, blue light has a higher index of refraction (slower speed) than red light. This is why prisms spread white light into a rainbow with red on the outside and violet on the inside.

Because the three channels are independent waves each solving their own wave equation with a slightly different speed constant, they naturally bend at different angles when exiting the prism — exactly like real chromatic dispersion.

06 — The Light Source

Injecting White Light

The light beam is injected as a Gaussian-modulated plane wave during the first fraction of a second (u_time < 0.1).

float circleWave(vec2 point, vec2 center, float freq, float size) {
  float r = pow(point.x - center.x, 2.0) + pow(point.y - center.y, 2.0);
  float fade = exp(-r / 2.0 / pow(size, 2.0)) / size;
  return fade * cos(freq * point.x) * abs(sin(u_time * 3.0));
}

Breaking this down:

Gaussian Envelope

exp(-r² / 2σ²) — A bell-curve fade centered at (-0.4, 0.1) with σ = 0.02. This confines the source to a small spot, creating a beam rather than filling all space.

Plane Wave

cos(700 × x) — A high-frequency cosine along the X axis. The frequency of 700 sets the wavelength of the light. Because it multiplies the same value into all 3 channels, the source is white light.

Rotation

The coordinates are rotated by -0.2 radians (≈ -11.5°) before the cosine, angling the beam so it enters the prism at an angle — necessary for refraction to spread the colors.

Time Pulse

|sin(time × 3)| gently pulses the source on/off, and the time < 0.1 guard means the source only emits briefly — then the wave propagates freely forever.

07 — Ping-Pong Rendering

Double Buffering on the GPU

A GPU shader can't read from and write to the same texture simultaneously. The solution: ping-pong between two sets of textures.

Each call to logic() does two simulation steps (A→B then B→A). The update loop calls logic() 4 times per frame, giving 8 wave equation steps per rendered frame. This sub-stepping makes the wave propagate fast enough to be visually interesting.

function update() {
  for(let i = 0; i < 4; i++) logic(); // 4 × 2 = 8 steps
  render(); // display the result
  window.requestAnimationFrame(update);
}
08 — Boundary Conditions

Absorbing Borders

Without boundary handling, waves would reflect off the edges of the texture, creating unwanted interference. The border() function creates an absorbing boundary:

void border(inout vec3 amp, inout vec3 vel) {
  float falloff = 1.0 - clamp(box(v_texcoord - 0.5, vec2(0.45)), 0.0, 1.0);
  amp *= falloff;
  vel *= falloff;
}

This uses a box SDF centered on the texture. Within a 0.45 half-size box (i.e., the inner 90%), falloff = 1.0 (no damping). Near the edges, falloff smoothly drops to 0, killing both amplitude and velocity — the wave simply fades away instead of bouncing back.

09 — Final Rendering

From Exposure to Pixels

The render shader reads the accumulated exposure texture (not the raw wave amplitude) and tone-maps it to produce the final image.

1

Clamp & Square

col = clamp(col, 0, 1); col *= col; — Clamps values to [0,1] and applies a gamma-like squaring that deepens contrast and makes the dark areas darker.

2

Blue-to-Violet Correction

Real violet isn't just blue — it has red mixed in. The shader computes how "blue-dominant" a pixel is and adds red to simulate violet:

float blue = col.b / (col.r + col.g + col.b);
blue = pow(blue, 4.0) * col.b;
col.r += blue; // add red to blue-heavy areas → violet!
3

Prism Tint

The glass region is tinted with a subtle blue-gray color using the wall map:

col += vec3(0.2, 0.25, 0.3) * sqrt(1.0 - d); // d=0.6 inside prism
10 — Complete Pipeline

Putting It All Together

Summary The simulation is a 2D wave equation on a cellular automata grid, where each cell stores amplitude and velocity for 3 color channels independently. A triangular prism slows waves to 60% speed, with each channel (R, G, B) receiving a slightly different speed offset. This tiny speed difference causes the wavefronts to bend at different angles — chromatic dispersion — producing a rainbow. The accumulated energy over time creates the glowing light trails you see on screen.
